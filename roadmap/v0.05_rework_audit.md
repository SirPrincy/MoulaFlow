# Audit V0.05 — Rework & Refonte (Moula Flow)

Date: 2026-04-07

## TL;DR
Pour une v0.05 solide, il faut **éviter une refonte globale** et faire une v0.05 orientée stabilisation + découpage:
1. **Rework prioritaire**: gros écrans/widgets, i18n, settings bootstrapping, cohérence UX formulaires.
2. **Refonte ciblée** (seulement 1 bloc): navigation/app shell.
3. **Pas de réécriture totale** de la data layer Drift/Riverpod: la base est déjà exploitable.

---

## Ce qui mérite un rework (important, mais pas “tout refaire”)

### 1) Découper les fichiers trop volumineux
Actuellement plusieurs composants sont très gros (risque de régressions + maintenance lente):
- `lib/pages/category_overview_page.dart` (~1538 lignes)
- `lib/widgets/dashboard_cards.dart` (~1029 lignes)
- `lib/settings_page.dart` (~853 lignes)
- `lib/widgets/transaction_form.dart` (~783 lignes)
- `lib/home_page.dart` (~746 lignes)
- `lib/models.dart` (~733 lignes)

✅ Action v0.05:
- Extraire des sous-widgets + view models par domaine (dettes, projet, budgets, formulaire transaction).
- Viser des fichiers <= 250-300 lignes pour les écrans complexes.

### 2) Internationalisation incomplète
On trouve encore beaucoup de textes en dur FR/EN dans les pages/widgets, ce qui casse la promesse l10n.

✅ Action v0.05:
- Migrer tous les labels/snackbars/dialogs vers `AppLocalizations`.
- Ajouter un check CI simple bloquant les `Text('...')` hardcodés hors fichiers l10n.

### 3) Initialisation des préférences trop centralisée dans `main.dart`
Le chargement des settings est fait unitairement avant `runApp` et injecté via overrides.
Ce pattern fonctionne, mais grossit vite et complique les évolutions de boot flow.

✅ Action v0.05:
- Introduire un `AppSettingsState`/`AppBootstrapState` unique (AsyncNotifier).
- Garder `main.dart` minimal (bootstrap + fallback loading UI).

### 4) Harmonisation formulaires & validation
Les formulaires sont riches mais dispersés, avec validations/snackbars hétérogènes.

✅ Action v0.05:
- Créer une couche commune (`AppFormField`, `AppValidators`, messages d’erreur localisés).
- Uniformiser les règles montant/date/wallet/catégorie.

---

## Ce qui peut être refait entièrement (refonte ciblée)

### Navigation / App Shell
Oui, c’est le seul vrai candidat “à refaire entièrement” côté structure:
- La roadmap mentionne déjà une migration GoRouter.
- Vu le nombre d’écrans/flows (wizard, lock, dashboard, dettes, projets, réglages), un routeur déclaratif simplifiera:
  - redirections,
  - deep links,
  - navigation desktop/mobile cohérente,
  - testabilité des flows.

✅ Action v0.05:
- Refondre la navigation avec `GoRouter` + guards (onboarding / app access / lock).
- Conserver les pages métiers et migrer progressivement (pas big-bang côté UI).

---

## Ce qu’il NE FAUT PAS refaire entièrement maintenant

### Data layer Drift + repositories
La base (offline-first, migrations, intégrité, backups) est déjà en place dans la doc projet.
Réécrire cette couche maintenant serait coûteux et peu rentable par rapport au gain.

✅ Action v0.05:
- Garder l’existant, renforcer avec tests migrations/backup et conventions repo.

---

## Plan v0.05 proposé (pragmatique)

### Sprint A (stabilité)
- Migration i18n des textes en dur critiques.
- Quick wins lint/deprecations.
- Standardisation validateurs/form components.

### Sprint B (architecture UI)
- Découpage `category_overview_page` + `dashboard_cards`.
- Isolation de `settings_page` par sections indépendantes.

### Sprint C (refonte ciblée)
- Introduction GoRouter + guards.
- Migration progressive des routes existantes.

---

## Verdict
- **À rework:** oui, clairement (UI monolithique, i18n, bootstrap settings, formulaires).
- **À refaire entièrement:** oui, mais **uniquement la navigation/app shell**.
- **À ne pas refaire entièrement:** data layer Drift/Riverpod (à consolider, pas à réécrire).
